## C. 金額計算規則建議（MVP）

### C.1 MVP 計算規則

| 計算項目            | MVP 規則                              | 備註                             |
| ------------------- | ------------------------------------- | -------------------------------- |
| 訂單小計            | `unitPrice × quantity`                | 目前已實作（totalPrice）         |
| 運費                | 後台手動輸入，預設 0                  | 目前已實作                       |
| 折讓金額            | 後台手動輸入絕對金額，不可為負        | 需新增 DB 欄位 `discount_amount` |
| 折數                | **不建議 MVP 同時做**，折讓金額已足夠 | 折數 + 折讓金額同時開放易出錯    |
| 折讓備註            | 後台輸入，僅後台可見                  | 需新增 DB 欄位 `discount_note`   |
| 訂單總額            | `小計 + 運費 - 折讓金額`              | formatOrder 邏輯需修改           |
| 購物金折抵          | **不建議本階段做**，需要會員系統      | 目前無會員 / 購物金資料表        |
| 購物金餘額          | **不建議本階段做**                    | 同上                             |
| 待收金額            | `max(訂單總額 - 已收金額, 0)`         | 目前已有但未含折扣               |
| 退款 / 取消影響金額 | **建議本階段不做**，手動記錄即可      | 取消訂單已有狀態，金額不自動調整 |
| 毛利計算            | **不建議本階段做**                    | 需先建立商品成本欄位             |

### C.2 MVP 折扣模型

```
訂單總額 = (unitPrice × quantity) + shippingFee - discountAmount
待收金額 = max(訂單總額 - paidAmount, 0)
```

- `discountAmount`：後台手動輸入，不可為負，不可超過小計 + 運費
- 折數與折讓金額**不可同時使用**（MVP 階段只開放其中一個，建議固定用金額）
- 購物金折抵：**等待會員系統，本階段不做**

### F.2 MVP 刪除 / 封存建議

| 建議項目                                          | 理由                                                           |
| ------------------------------------------------- | -------------------------------------------------------------- |
| **不建議永久刪除（hard delete）**                 | 訂單是財務憑證，刪除後無法追查                                 |
| **建議先做封存（soft delete，`archivedAt`）**     | 保留資料、不影響正常查詢、可恢復                               |
| **刪除 / 封存前需二次確認**                       | 與取消訂單一樣需要 AlertDialog                                 |
| **封存後預設不顯示於訂單列表**                    | 需在 GET /stores/:storeId/orders 加 `WHERE archivedAt IS NULL` |
| **封存後 CSV / picking / shipping list 也不包含** | 需同步修改三個查詢                                             |
| **不建議本階段做封存列表**                        | MVP 先做「封存但不顯示」，有需要再加封存列表查詢               |
| **不建議實作永久刪除 API**                        | 保留 archivedAt soft delete 機制即可                           |

## H. 風險與測試清單

### H.1 金額調整風險

| 風險                   | 說明                                                          | 建議處理                                             |
| ---------------------- | ------------------------------------------------------------- | ---------------------------------------------------- |
| 金額錯算               | `totalPrice` 目前由後端計算，加入折扣後需同步修改 formatOrder | 修改後需加 unit test                                 |
| 折扣負數               | 若 discountAmount 可為負，訂單總額可能超過原價                | API 驗證：discountAmount >= 0                        |
| 訂單總額低於 0         | 折讓金額可能超過小計 + 運費                                   | API 驗證：discountAmount <= totalPrice + shippingFee |
| 折數與折讓金額同時使用 | 兩者並存會造成計算混亂                                        | MVP 只保留其中一個欄位                               |
| 購物金超扣             | 無會員系統，不知道餘額上限                                    | **不建議本階段做購物金功能**                         |
| 舊訂單顯示不一致       | 新增折扣欄位後舊訂單 discountAmount = null                    | formatOrder 需處理 null（視為 0）                    |

### H.2 PDF 風險

| 風險                 | 說明                                   | 建議處理                              |
| -------------------- | -------------------------------------- | ------------------------------------- |
| PDF 金額與後台不一致 | 若 PDF 獨立計算金額，容易與後台不同步  | PDF 資料從 API 取得，不自行重算       |
| 中文字型問題         | jsPDF 等套件需嵌入字型才能顯示中文     | MVP 優先用 window.print，不需額外字型 |
| iOS 列印不相容       | iOS Safari 對 window.print 支援有限    | 需實機測試，提前規劃降級方案          |
| 個資保護             | PDF 含買家電話、地址，不可讓第三方截取 | PDF 只在後台產生，不對外公開 URL      |

### H.3 刪除 / 封存風險

| 風險                                 | 說明                                          | 建議處理                                          |
| ------------------------------------ | --------------------------------------------- | ------------------------------------------------- |
| 刪除後無法追查                       | 永久刪除無法恢復                              | **不做永久刪除，只做 archivedAt soft delete**     |
| CSV 匯出包含封存訂單                 | 目前 CSV 查詢無過濾，未來加入封存後需同步修改 | 修改 CSV 查詢排除 archivedAt IS NOT NULL          |
| picking / shipping list 包含封存訂單 | 同上                                          | 同步修改三個查詢                                  |
| 封存誤操作                           | 老闆誤封存仍在處理的訂單                      | 封存前二次確認彈窗（AlertDialog），與取消訂單一致 |
| 誤把 cancelled 當封存                | 目前 cancelled 是狀態，不是刪除               | 文件說明兩者差異，避免混淆                        |

### H.4 備註風險

| 風險                | 說明                                             | 建議處理                                                           |
| ------------------- | ------------------------------------------------ | ------------------------------------------------------------------ |
| 客服誤解備註內容    | internalNote 是後台內部備註，不能讓買家看到      | 確認公開訂單頁（PublicOrder.tsx）不顯示 internalNote / paymentNote |
| 備註在 CSV 中被洩漏 | 目前 CSV 匯出有意排除 internalNote / paymentNote | 不修改此行為                                                       |

## I. MVP 建議結論

### I.1 最先應做的功能

**建議優先順序：**

1. **折讓金額（Step 8I → 8J → 8K）**：高頻需求，折讓是代購系統核心場景（團購折讓、活動折扣）
2. **封存訂單（Step 8O → 8P）**：資料安全性，避免老舊訂單干擾清單，且風險較刪除低
3. **PDF 列印（Step 8M → 8N）**：用 window.print 可快速完成，提升操作專業感

### I.2 哪些功能不應在同一個 PR 做

| 組合                    | 建議                                    |
| ----------------------- | --------------------------------------- |
| 金額調整 + PDF + 刪除   | 禁止同 PR，功能不相關，scope 過大       |
| DB migration + UI 施工  | 建議分 PR（先 migration，再 UI）        |
| 折讓金額 + 購物金       | 禁止同 PR，購物金依賴尚不存在的會員系統 |
| 封存功能 + CSV 匯出修改 | 建議同 PR，兩者邏輯強耦合               |

### I.3 哪些功能需要 DB migration

| 功能                                      | 需要 migration               |
| ----------------------------------------- | ---------------------------- |
| 折讓金額（discountAmount / discountNote） | **是**                       |
| 封存（archivedAt）                        | **是**                       |
| 備註正式化                                | **否**（欄位已存在）         |
| PDF（用 window.print）                    | **否**                       |
| 毛利計算                                  | **是**（需新增商品成本欄位） |

### I.4 哪些功能可以先只做 UI

| 功能                     | 可先只做 UI                                    |
| ------------------------ | ---------------------------------------------- |
| PDF 列印（window.print） | **是**（前端 CSS print 即可，不需後端 API）    |
| 折讓金額顯示（唯讀）     | **否**（需先有 DB 欄位才有資料可顯示）         |
| 封存按鈕（樣式）         | 可先做 UI 骨架，但 API 未完成前按鈕應 disabled |

### I.5 哪些功能必須等會員 / 購物金系統

| 功能                    | 依賴                                                |
| ----------------------- | --------------------------------------------------- |
| 購物金折抵              | **必須等**：需要會員 ID、購物金餘額資料表、扣點 API |
| 購物金餘額顯示          | **必須等**：同上                                    |
| 自動退款 / 退款到購物金 | **必須等**：同上                                    |

**購物金相關功能本階段一律不做。**
