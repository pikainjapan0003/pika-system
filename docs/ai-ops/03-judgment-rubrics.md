# 03 判斷力外化：判準、步驟、正例、反例、最低驗收

> 讀者：Sonnet/Haiku 級模型。每條都可機械執行，不要求「自行判斷」。

## 1. 何時該升級模型

**判準**：出現以下任一訊號就升級，不要再重試同級模型：
- 同一子任務同級模型已失敗兩次（haiku 級一次即升）。
- 任務含商業取捨、產品品味、客人體驗判斷（例如「這個下單流程順不順」）。
- 任務要求設計新的成本公式或改動既有金額公式（不是照 spec 實作）。
- 失敗原因看不懂（錯誤訊息與改動無法對上）。

**操作步驟**：1) 整理失敗軌跡（輸入、輸出、錯在哪，各 ≤10 行）；2) 帶軌跡升級或回報使用者；3) 升級後解出的模式寫成 checklist 降回便宜模型套用。
**正例**：「Sonnet 兩次把 discountAmount 算進運費，附兩次 diff，請 Opus 判斷 schema 語意。」
**反例**：換措辭對同級模型重試第三次。
**最低驗收**：升級訊息內含失敗軌跡；沒有軌跡的升級 = 無效升級。

## 2. 何時算真的完成

**判準**：五項全過才叫完成：
1. 目標檔案已寫入。
2. read-back 看過新內容（重新開檔，不是憑 Write 成功訊息）。
3. 程式碼：`pnpm run typecheck` 通過；有對應 `.test.mjs` 的要跑（無法跑要寫原因，例如無 DATABASE_URL）。
4. 涉及成本／金額：至少一筆樣本資料手算比對（見 §8）。
5. 回報包含：檔案路徑、驗證方式、殘留風險。

**正例**：「已更新 docs/ai-ops/02-model-orchestration.md，read-back 確認升級規則存在；無程式碼變更；殘留風險是尚未確認實際可用模型名稱。」
**反例**：「我已經完成模型調度守則。」（無路徑、無驗證）
**最低驗收**：回報缺任一必含項 → 視為未完成，補完才能結案。

## 3. 何時該停下來問使用者

**判準**（任一命中就停）：
- 要動 L0 檔（CLAUDE.md、docs/order-step* 既有 spec 的定案內容）。
- 要刪除或覆蓋不是本 session 建立的檔案，且內容與描述不符。
- 成本公式的「商業參數」沒有依據（例如服務利潤率抓多少、衝量價怎麼定）。
- 需要憑證／付費／對外發布（push、部署、寄信、開 API key）。
- dev-handoff worker 身份未指定（CLAUDE.md 硬規定）。

**操作步驟**：停下前先把已完成部分落檔，再用一則訊息列出「問題＋你建議的選項＋各選項後果」。
**正例**：「利潤率 spec 沒定義；建議先用 Sheet 既有值（待實讀），或請你給暫定值。已把其餘欄位遷移完成並落檔。」
**反例**：自行假設利潤率 15% 寫進程式且不標註。
**最低驗收**：問題訊息含建議選項；不是開放式「請問怎麼辦」。

## 4. 什麼訊號代表方向錯了（換路，不是重試）

**判準**：
- 修 A 壞 B、修 B 壞 A，連續兩輪。
- 為了通過驗證而開始改驗證本身（改測試期望值、放寬 schema）。
- 方案需要的前提查不到（API 不存在、欄位不存在），卻在想辦法「繞過」。
- diff 越改越大但驗收項通過數沒增加。

**操作步驟**：1) 停手；2) 一句話寫下原路的前提假設；3) 檢查哪個前提是假的；4) 換路（縮小範圍／改人工輸入／回報做不到）。
**正例**：gogo.gs 沒 API → 換路做手動輸入欄位，而不是寫 HTML 爬蟲。
**反例**：測試一直紅，就把測試的期望金額改成程式算出來的值。
**最低驗收**：換路決定要寫進回報與（若影響長期）05 Lessons Log。

## 5. 品質底線怎麼驗

**判準**：文件類 = read-back + 引用路徑抽查 2 處是否真實存在；程式類 = typecheck + 相關測試 + 一筆樣本實跑；生成物（lib/api-zod 等）= 只能由 codegen 產生，不可手改。
**操作步驟**：驗證由 fresh-context agent 或獨立流程做（02 檔 §6），修改者不可自驗高風險改動。
**正例**：改完 orders route 後，另派 agent 用一筆樣本 POST 比對總額。
**反例**：「我檢查過我自己的 diff，沒問題。」
**最低驗收**：回報寫明「誰驗的、怎麼驗的」。

## 6. 什麼任務可以自動做 / 不能自動做

**可以自動**（不必問）：讀檔調查、寫 docs/ai-ops 草稿、加測試、typecheck、格式化、在新檔實作已有 spec 的功能、本地 commit（kebab-case）。
**不能自動**（必問或必禁）：push GitHub（CLAUDE.md 禁止）、動 production DB、對 production 跑測試、改金額公式的商業參數、刪既有資料檔（data/）、對外送訊息、把憑證寫進 repo。
**正例**：自動新增 shippingFee 的單元測試 → 可。
**反例**：自動把 shippingFee 費率表從 60 改 65「因為看起來合理」→ 禁；費率是商業參數。

## 7. 什麼改動屬高風險

**判準**：碰到以下任一即高風險，適用 §2/§5 加強版（獨立驗證＋樣本手算）：
- 檔案：`shippingFee.ts`、`routes/orders.ts`、`routes/public.ts`、`lib/db/src/schema/orders.ts`、未來的 cost/pricing 模組。
- 欄位：unitPrice、totalPrice、paidAmount、discountAmount、任何未來成本欄位。
- 行為：訂單狀態機轉移、物流匯入寫 DB、客資（姓名電話地址）處理。

**反例（低風險誤判成高風險）**：改 README 錯字不需要獨立驗證。
**最低驗收**：高風險改動的回報必附樣本計算過程（輸入 → 每步 → 輸出）。

## 8. 成本計算如何驗

**背景**：成本模組目前不存在（2026-07-07），未來從 Sheet 遷移。概念欄位：日圓成本、匯率、重量、單件交通費、航運運費、信用卡手續費、百貨手續費、空運選擇、目標/最終售價、淨利、服務利潤率、毛利、衝量利潤、客人心目中差價。**範圍注意：老闆已裁決成本 Sheet 僅用於「交通成本分攤」線（見 12 檔檔頭）；其餘成本概念屬另一條線，未解凍前不做。**

**操作步驟（任何成本程式改動）**：
1. 取一筆真實樣本（Sheet 匯出列或既有訂單）。
2. 手算：日圓成本×匯率 → 加各手續費 → 加交通分攤與運費 → 得成本；售價−成本 = 毛利。逐步寫出。
3. 跑程式同一筆輸入，逐欄比對；差異 >0.5 元（或 >0.1%）即失敗。
4. 邊界各測一筆：重量 0／未填、匯率未填、手續費 0、免運。
5. 未填欄位的行為必須是「擋下並提示」，不可默默當 0 算出低成本。

**正例**：「樣本：¥1,200×0.215=258.0，＋刷卡 1.5%＝261.9，＋交通分攤 12.0＝273.9；程式輸出 273.9，一致。」
**反例**：「成本函式測試都綠的。」（測試是自己寫的期望值，沒有對照 Sheet 手算）
**最低驗收**：回報含至少一筆逐步手算。

## 9. 物流與 7-11 店鋪選擇如何驗

**產品底線**：客人快速選店、少輸入、選店結果回填訂單；老闆後台直接看到門市店號店名。不是地圖玩具。

**操作步驟**：
1. 改動後跑 `cvs.route.test.mjs` 與相關 logistics 測試（完整指令見 01 檔速查：需 `--experimental-test-module-mocks` flag 與 DATABASE_URL，非 production；runner 以測試檔頭註解為準）。
2. 手動驗一次完整流程：公開下單頁 → 選 7-11 → 選店 → 回到訂單 → 門市資訊出現在訂單與後台。
3. 運費檢查：每種物流方式（7-11／全家／黑貓／郵局／面交）各下一筆，總額 = 商品小計＋該方式運費（費率表 `shippingFee.ts`）。
4. 匯入類改動：先跑 dry-run worker（`multiProviderDryRunWorker.ts`），比對 dry-run 輸出再開 controlled write。

**反例**：只驗了選店 UI 會開，沒驗店號有沒有寫回訂單。
**最低驗收**：回報列出驗過的物流方式與各自總額。

## 10. Google Sheet 欄位如何遷移成系統規則

**前提**：Sheet 匿名讀取 401；必須 (a) service account 實讀（SA 金鑰在使用者本機，勿入 repo）或 (b) 請使用者匯出 CSV。**沒有 (a)(b) 就不得開工。**

**操作步驟**：
1. 實讀 Sheet，逐欄記錄：欄名、範例值、推測用途、公式（看得到才寫，看不到標「未確認」）。落檔 `docs/ai-ops/`（建議 `10-cost-sheet-mapping.md`）。
2. 每個欄位分類：輸入值（老闆填）／參數（匯率、費率）／計算值（公式）。
3. 計算值先寫成純函式＋以 Sheet 實際數字做 test fixture（Sheet 的一列 = 一個測試案例）。
4. 公式與 Sheet 結果對不上時：**以 Sheet 為準回報差異**，不可假設 Sheet 錯了自行修正——Sheet 是現有商業邏輯的證據。
5. 遷移後保留人工覆寫欄位（臨時改價、重量估錯、特殊物流都是常態）。

**反例**：嫌 Sheet 亂，直接設計「更合理的」成本模型。
**最低驗收**：每個計算欄位至少一個來自 Sheet 真實列的通過測試。

## 11. 油價整合（gogo.gs 型網站）如何判斷可行

**判準（四關全過才可整合，任一不過就走手動輸入）**：
1. 有官方 API 或官方資料下載（文件頁可引用 URL）。
2. ToS/robots 允許程式化取用。
3. 不需要執行 JS／過人機驗證就能取得資料。
4. 失敗可偵測（取不到值時系統會告警，不會默默用舊值）。

**2026-07-07 現況**：gogo.gs 首頁 200（需 UA），未發現官方 API（未確認；確認方法：查 gogo.gs 網站頁尾／help 是否有 API 文件、搜「gogo.gs API 公式」）。→ **目前走手動輸入。**

**手動輸入替代方案（設計基準）**：
- 成本暫存區保留 `油價(元/L)` 與 `旅程成本` 手動輸入欄；UI 附 gogo.gs 連結供老闆查價。
- 單件交通分攤 = 旅程成本 ÷ 該趟商品件數（與 Sheet 現行邏輯核對後才定案，見 §10）。
- 油價欄位帶「上次更新日期」，超過 30 天在 UI 提示過期，但不擋流程。

**反例**：寫 regex 爬 gogo.gs HTML、假 UA 繞擋、無告警。
**最低驗收**：任何油價整合提案必須逐條回答上面四關，附證據 URL。

## 12. harness 極限（誠實條款，全 rubric 通用）

拆解、驗證、多樣本評審可以補執行品質。但**模糊題、產品品味、商業取捨、客人體驗判斷不能靠弱模型補**——遇到就 §1 升級、找第二意見、多方案評審，或明說做不到。查不到的事標：未確認／原因／下次怎麼確認／目前安全假設。不要編造。
