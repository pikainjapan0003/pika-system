# 購物金規格草案

- 日期：2026-07-18
- 狀態：**設計草案；所有金額細則均待老闆拍板，嚴禁先行實作**
- 已拍板方向（Q23）：先完成資料模型與流程草案，再回答題卡。

## 現況盤點

### 客戶

- `customers` 已以 `(store_id, code)` 唯一識別同店客戶，並有 `general / vip / wholesale / partner` 四層等級。
- 客戶可沒有手機；因此購物金不能以手機當唯一帳戶鍵。
- 訂單的 `customer_id` 可為空且刪客戶時 `SET NULL`；購物金交易若涉及對帳，不可跟著客戶硬刪消失。

### 訂單與付款

- 訂單已有成交 `unit_price`、`total_price`、`discount_amount`、`paid_amount`、付款狀態與永久成本／毛利快照。
- 四層價格先決定成交單價；購物金是成交價之後的付款／折抵，不應改寫商品分級價格。
- 目前沒有購物金欄、購物金流水、保留額或退款回沖欄。

## 資料模型提案

### 不建議：只在 customers 加一個餘額欄

單一餘額欄雖簡單，但無法回答「誰在何時因哪張訂單加減多少」、無法可靠退款，也難以處理兩個請求同時扣款。它也讓任何手動 UPDATE 都變成無歷史紀錄的帳務變更。

### 建議：不可變交易流水為主

新增概念表 `store_credit_transactions`（名稱暫定）：

| 欄位概念                    | 用途                                   | 設計限制                                 |
| --------------------------- | -------------------------------------- | ---------------------------------------- |
| `id`                        | 流水唯一鍵                             | 不可重用                                 |
| `store_id`                  | 店鋪隔離                               | 必填；所有查詢先綁店鋪                   |
| `customer_id`               | 所屬客戶                               | 必填；客戶軟刪後流水仍保留               |
| `order_id`                  | 關聯訂單                               | 可空；退款／折抵時應填                   |
| `type`                      | 儲值、折抵、退款回沖、手動調整、到期等 | 精確 enum 待拍板                         |
| `amount_twd`                | 有方向的台幣變動                       | 精度、是否只允許整數、正負號慣例皆待拍板 |
| `status`                    | 暫存、已入帳、取消等                   | 是否需要保留額機制待拍板                 |
| `idempotency_key`           | 防止同一訂單重複扣／退                 | 同店唯一                                 |
| `reason_code`／`note`       | 可稽核原因                             | 不存明文個資；自由文字是否允許待拍板     |
| `created_by`                | 操作者 opaque ID                       | 不存 token 或登入密鑰                    |
| `created_at`／`reversed_at` | 帳務時間                               | 流水不覆寫；修正以新反向流水處理         |

餘額應由「已入帳流水」精確加總得出。若效能需要，可另做 cache／materialized balance，但它只能是可重建投影，不能成為唯一帳本。

### 必要資料庫防線

- 全程 PostgreSQL `numeric`＋既有 ExactDecimal；禁 JavaScript float 累加。
- 同店 idempotency key 唯一，避免重送造成雙扣／雙退。
- 任一扣款與訂單寫入必須在同一資料庫交易內完成。
- 餘額是否允許負數待拍板；若拍板禁止，扣款時需鎖客戶帳本或採原子條件更新，不能只在 UI 先查餘額。
- 已入帳流水不可 UPDATE 金額或 DELETE；錯誤以反向流水修正。

## 與訂單金額的互動

以下欄位與公式尚未拍板，現階段只定義應回答的問題：

1. `unit_price` 仍是四層定價解析後的成交單價，不因購物金改變。
2. `total_price` 是商品、運費、折扣後哪一個階段的金額，必須先確認現行契約；不可直接假設購物金要從它扣除。
3. 建議訂單新增「購物金折抵快照欄」與「應收現金／其他付款金額欄」，但欄名、精度及公式待拍板。
4. 成本與毛利快照是成交經濟結果；購物金究竟視為付款工具、行銷折讓或兩者依來源不同，會影響毛利／會計，必須由老闆決定，禁止自行改現有毛利公式。
5. 購物車訂單同樣只在訂單層扣一次，不把同一筆折抵任意拆進每個 item，除非另有經審核的分攤規則。

## 建議流程（語意待拍板）

### 儲值／發放

1. owner 選客戶與來源原因。
2. 顯示變動前後餘額預覽。
3. 二次確認後新增一筆正向流水並記 audit。
4. 不允許直接覆寫「餘額」。

### 下單折抵

1. 客戶必須已被識別為特定 `customer_id`；匿名單不可猜測歸戶。
2. 計算訂單成交價後，顯示可用餘額與本單允許折抵上限。
3. 送單交易內：驗餘額／額度、建立訂單、建立折抵流水、寫訂單折抵快照，一起 commit；任一失敗全部 rollback。
4. 重送同一請求由 idempotency key 回傳原結果，不再扣一次。

### 取消／退款

- 不修改原折抵流水；新增一筆具 `reversal_of` 關係的回沖流水。
- 取消訂單是否自動回沖、退現金或退購物金、過期額度是否恢復，皆待拍板。
- 已完成、部分退款、跨月退款與人工退款需有明確狀態機後才能實作。

## 權限、個資與 audit

- 管理端操作全部 `requireAuth`＋`verifyStoreOwner`；查詢從 `store_id` 隔離。
- 客人端若顯示餘額，只能在已驗證身分的頁面；不能靠可轉傳的 publicToken 直接看完整帳戶流水。
- 後台客戶頁預設只顯示必要摘要；流水不得附帶明文姓名、手機、地址。
- audit log 記動作、店鋪、操作者 opaque ID、客戶／訂單 opaque target、流水 ID、時間；不記 token、卡號、匯款資訊或自由輸入個資。
- 明文匯出購物金流水若未來需要，另走二次確認＋明文 header gate；本草案不自動納入現有客戶 CSV。

## 與客戶等級的關係

- 四層價格決定商品成交單價；購物金預設不改變客戶 tier。
- 是否只有特定 tier 可用、發放比例是否依 tier 不同、升降級是否影響既有餘額，全部待拍板。
- 保守建議：第一版所有已建檔客戶都可有帳戶，tier 只影響價格、不影響既有購物金，避免把兩套規則耦合；仍需老闆核准。

## 待拍板題卡

### Q1 購物金從哪裡來？

- A. 僅老闆手動發放／調整。
- B. 客人儲值＋老闆手動發放。
- C. 再加消費回饋自動發放。
- **保守預設：A。** 先避開金流與自動回饋公式；實際選項待拍板。

### Q2 每單折抵上限？

- A. 可折到應付 0。
- B. 訂單金額固定百分比。
- C. 固定台幣上限。
- **保守預設：待拍板前不開折抵。** 任一數字都是新金額規則，不能由開發者決定。

### Q3 取消／退款怎麼回沖？

- A. 原使用多少購物金就全數退回。
- B. 依實際退款比例回沖。
- C. 人工決定退現金或購物金。
- **保守預設：人工審核後建立反向流水。** 自動比例與混合退款需另有公式。

### Q4 有沒有使用效期？

- A. 永久有效。
- B. 每筆發放各自到期。
- C. 統一年度到期。
- **保守預設：A。** 到期會增加批次扣除、通知與回沖複雜度；實際仍待拍板。

### Q5 是否禁止負餘額？

- A. 永遠禁止。
- B. 僅 owner 可手動調成負數。
- **保守預設：A。** 資料庫／交易層原子阻擋，不只前端阻擋；待拍板。

### Q6 誰能手動調整？

- A. 只限店主。
- B. 店主與具特定技能／角色的員工。
- **保守預設：A。** 現有系統以 owner 防線最清楚；角色模型未成熟前不擴權。

### Q7 購物金算付款還是折扣？

- A. 付款工具，不改成交毛利。
- B. 行銷折扣，要影響毛利。
- C. 依發放來源分類。
- **無保守代選。** 這會直接改訂單、毛利與會計語意，必須由老闆明確拍板。

### Q8 匿名客人能不能使用？

- A. 必須選定 customer 才能使用。
- B. 用手機歸戶。
- **保守預設：A。** 手機可空且不是可靠唯一鍵，不應自動猜客戶。
