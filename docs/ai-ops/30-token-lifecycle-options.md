# 公開 Token 生命週期選項

盤點日：2026-07-19  
範圍：商品 `shareToken` 與訂單 `publicToken` 的撤銷、輪替、有效期。只做設計與題卡，不改程式。

## 1. 現況事實

| Token                  | 用途                                       | 產生方式                                                                                         | 熵與格式                              | 現有生命週期                                                                                                                                         |
| ---------------------- | ------------------------------------------ | ------------------------------------------------------------------------------------------------ | ------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------- |
| `products.share_token` | 客人開商品頁、送出單品訂單、購物車指定商品 | 建商品時 `randomBytes(12).toString("hex")`（`artifacts/api-server/src/routes/products.ts:68`）   | 12 bytes＝96 bits，24 個十六進位字元  | DB unique（`lib/db/src/schema/products.ts:36`）；沒有輪替／到期 API。商品 `is_active=false` 時公開頁回 404（`public.ts:114-125`），但 token 本身未改 |
| `orders.public_token`  | 客人查單與更新付款末五碼                   | 公開單 `randomBytes(16).toString("hex")`（`public.ts:174,335`）；後台單同規格（`orders.ts:140`） | 16 bytes＝128 bits，32 個十六進位字元 | DB unique（`lib/db/src/schema/orders.ts:87`）；碰撞會重試（`public.ts:291-300,479-488`）；沒有撤銷、輪替或到期欄                                     |

兩者都是 bearer link：知道完整網址的人即可使用公開功能。公開頁已設 `Referrer-Policy: no-referrer`，降低 token 因 Referer 流到外站的機會；但轉貼、截圖、瀏覽器紀錄與聊天備份仍可能留下連結。

## 2. 撤銷選項

### A. 立即失效旗標（保守、建議）

- 商品：沿用 `is_active=false` 作整件商品下架；若要只撤銷舊連結，仍需搭配輪替。
- 訂單：新增 `public_access_revoked_at`，公開查單與付款末五碼端點一律先檢查；撤銷後回 404，避免暴露訂單是否存在。
- 操作：店主二次確認、寫 `audit_logs`，畫面明示「客人的舊連結會立刻失效」。
- 優點：規則清楚、可立即止血；缺點：客人需要店主另行提供新連結或人工服務。

### B. 只靠換 token

- 更新 token 即使舊連結失效，不另存 revoked 狀態。
- 優點：欄位較少；缺點：無法區分「從未分享」與「因事故撤銷」，稽核與客服判讀較差。

### C. 撤銷清單／token 子表

- 一筆商品或訂單可有多個 token，每個有狀態與時間。
- 優點：完整稽核、可管理多連結；缺點：資料模型與查詢複雜，現階段需求不足。

## 3. 輪替選項

### A. 店主手動立即輪替（保守、建議）

- 點「重設公開連結」後產生同等熵的新 token，舊 token 同交易失效。
- 二次確認＋audit；回應只顯示新連結，不把 token 寫進 audit log。
- 商品輪替會讓購物車與舊分享網址失效；訂單輪替要把新查單網址重新交給客人。

### B. 定期自動輪替

- 每 N 天自動換 token。
- 優點：縮短長期外洩窗口；缺點：大量正常客人連結無預警失效，客服成本高，不建議作預設。

### C. 有重疊期的版本輪替

- 新舊 token 同時有效一段時間，再停用舊 token。
- 優點：體驗平滑；缺點：事故時不能立即止血，且需要 token 子表與排程。

## 4. 有效期選項

### A. 不自動到期（保守預設）

- 維持目前客人連結長期可用，但補上手動撤銷／輪替。
- 適合團購查單可能跨月、客人需要回看歷史的現況。

### B. 固定期限

- 商品連結可依活動結束日到期；訂單連結可在完成後 N 天到期。
- 必須先拍板 N、到期後客服流程、完成／取消單是否不同。

### C. 最後使用時間續期

- 每次開啟即延後期限。
- 優點：活躍客人不中斷；缺點：公開 GET 變成寫入、匿名流量可延長存活，稽核與濫用面較複雜。

## 5. 建議的最小安全路徑

1. 維持「不自動到期」，避免無預警破壞既有客人連結。
2. 商品與訂單各提供店主手動「立即輪替」；訂單另提供明確「撤銷公開查詢」。
3. 所有操作皆 requireAuth＋verifyStoreOwner、二次確認、audit，不在 audit 記 token 原文。
4. 公開端點對舊／撤銷 token 都回相同 404；限流與既有 allowlist 防線維持。
5. 未來若實際發生大量過期連結管理需求，再評估固定期限，不先上自動排程。

## 6. 老闆題卡

### 題 1：訂單查單連結是否要自動過期？

- A（建議）：不自動過期，只給店主撤銷／重設。
- B：訂單完成後 90 天到期。
- C：所有訂單建立後固定 180 天到期。
- 影響：B/C 會讓客人舊連結失效，需補客服與重新發連結流程。

### 題 2：商品分享連結重設時，舊購物車怎麼辦？

- A（建議）：舊 token 立即失效，購物車提示商品連結已更新，請重新加入。
- B：舊 token 保留 24 小時重疊期。
- C：商品連結不提供輪替，只能上下架。
- 影響：A 最能止血；B 體驗平滑但外洩窗口延長；C 最省工但事故處理能力最低。

### 題 3：訂單撤銷後可否恢復原連結？

- A（建議）：不可恢復原 token；需要時只產生新 token。
- B：可解除撤銷，原 token 再次有效。
- C：撤銷後永久不再提供公開查單。
- 影響：A 避免已外洩的舊網址死灰復燃，也保留客服補發能力。

## 7. 尚未拍板前的禁止事項

- 不得自行加入到期天數或自動輪替排程。
- 不得把 token 原文寫入 audit、log、CSV 或錯誤訊息。
- 不得以可猜的訂單 ID 取代 `publicToken`。
- 不得讓輪替改動訂單金額、快照或其他成交資料。
