# 04 派工 prompt 模板

> 用法：複製對應模板，填空，派給 subagent（模型選擇見 02 檔 §3）。缺「目標／驗收／回報格式」任一項的派工無效。
> 所有模板通用禁止事項：不得猜路徑與欄位、不得編造 API、查不到寫「未確認＋確認方法」、長產物落檔只回路徑。

---

## 模板 1：搜尋任務（Explore agent，便宜模型）

```text
任務背景：{為什麼要找}
目標與動機：{找什麼，找到後用來做什麼}
輸入資料：{起點路徑或關鍵字，例如 docs/order-step6*}
必查來源：{必須實際打開的檔案/目錄}
禁止事項：唯讀，不改任何檔；不確定不得猜。
驗收條件：
1. 每個結論附路徑:行號。
2. 找不到要明說「未找到」與已搜過的範圍。
驗收方式：主對話抽查 2 個路徑:行號是否真實存在。
回報格式：Markdown 條列，每點 = 結論＋路徑:行號＋風險；總長 <100 行。
升級條件：搜兩輪仍定位不到 → 回報已搜範圍，由主對話換關鍵字或升級模型。
落檔位置：{通常不落檔，回報即可；大型盤點落 docs/ai-ops/}
```

**範例（7-11 店鋪選擇現況）**：
> 任務背景：要改選店流程前先確認現況。目標：整理 7-11 選店從公開下單頁到訂單回填的完整資料流。輸入：`artifacts/shop-app/src/lib/cvs711.ts`、`artifacts/api-server/src/routes/cvs*`、`docs/order-step6*`。驗收：列出每一步的檔案:行號、店號店名回填到哪個欄位、有哪些測試覆蓋。回報 <100 行，只回結論。

## 模板 2：實作任務（general-purpose agent，中階模型起）

```text
任務背景：{spec 或需求來源，附檔案路徑}
目標與動機：{要做出什麼行為}
輸入資料：{要改的檔案清單；新檔給明確路徑}
必查來源：{動工前必讀的 spec/schema/既有實作}
禁止事項：不動生成物（lib/api-zod、lib/api-client-react）；不動費率等商業參數；不 push；不 stage dev-handoff/。
驗收條件：
1. pnpm run typecheck 通過。
2. {對應 .test.mjs 通過，或寫明無 DATABASE_URL 等原因＋替代驗證}
3. 高風險（金額/物流/客資）：附一筆樣本逐步手算（03 檔 §8）。
4. read-back 改過的每個檔。
回報格式：改了哪些檔（路徑:行號）、測試結果原文、風險、下一步；<80 行。
升級條件：同一失敗兩輪 → 帶失敗軌跡升級（02 檔 §5）。
落檔位置：程式碼進 repo；設計決策若有長期價值，一行進 05 Lessons Log。
```

**範例（新增商品成本計算函式）**：
> 背景：依 `docs/ai-ops/10-cost-sheet-mapping.md`（欄位對照，須先存在）。目標：`lib/` 新增純函式 `computeItemCost(input)`，輸入日圓成本/匯率/重量/交通分攤/各手續費，輸出成本與毛利。必查：Sheet 對照檔＋orders schema。驗收：typecheck 通過；用 Sheet 真實列做 3 個 fixture 測試通過；一筆逐步手算附在回報；未填欄位丟錯不默認 0。

## 模板 3：重構任務（general-purpose agent，中階模型）

```text
任務背景：{現況哪裡難維護}
目標與動機：{重構後的形狀；行為不得改變}
輸入資料：{目標檔案}
必查來源：{現有測試、呼叫端}
禁止事項：不改對外行為；不改測試期望值；不順手加功能。
驗收條件：
1. 重構前先跑一次測試記錄基準，重構後同套測試全綠。
2. typecheck 通過。
3. 呼叫端全部更新（grep 舊名稱 0 命中）。
回報格式：移動/改名對照表、測試前後結果、風險；<80 行。
升級條件：測試基準本來就紅 → 停，回報主對話，不得在紅測試上重構。
落檔位置：程式碼進 repo。
```

**範例（把成本公式抽成 domain service）**：
> 背景：成本計算散在 route 內（未來遷移後可能發生）。目標：抽到 `lib/` 純函式模組，route 只呼叫。禁止改任何數值行為。驗收：orders 相關測試前後全綠、grep 舊 inline 公式 0 命中。

## 模板 4：研究任務（general-purpose agent；結論影響商業決策時用高階模型）

```text
任務背景：{要評估什麼}
目標與動機：{研究結論會決定什麼}
輸入資料：{已知事實、已試過什麼}
必查來源：{官方文件/官網頁面，附 URL}
禁止事項：不得只憑訓練記憶下結論；每個結論要有本次實查證據；不可行就寫不可行，不得為了交差說可行。
驗收條件：
1. 逐條回答判準（例如 03 檔 §11 油價四關）。
2. 每條附證據 URL 或「未確認＋確認方法」。
3. 給出明確建議：做／不做／換替代方案。
回報格式：結論先行，證據表格，<100 行。
落檔位置：docs/ai-ops/（研究結果是長期資產，必落檔）。
升級條件：官方文件讀不出結論 → 標未確認並回報，不得腦補。
```

**範例（gogo.gs 油價整合可行性）**：
> 背景：油價影響交通成本分攤。目標：決定整合或維持手動輸入。必查：gogo.gs 網站是否有 API/資料下載頁、robots.txt、利用規約。驗收：逐條回答 03 §11 四關＋證據 URL；不可行就直接建議手動輸入方案。落檔 `docs/ai-ops/11-fuel-price-research.md`。

## 模板 5：審查任務（fresh-context agent，不可用寫程式的同一個 agent）

```text
任務背景：{被審查的改動/文件是誰、為了什麼做的}
目標與動機：{擔心什麼壞掉}
輸入資料：{diff 或檔案清單；審查者事前不知道實作細節，這是刻意的}
必查來源：{相關 spec、schema、測試}
禁止事項：不修改任何檔（只回報）；不因為「看起來合理」放行金額邏輯。
驗收條件：
1. 每個問題附路徑:行號＋失敗情境（什麼輸入會出什麼錯）。
2. 金額類：自己獨立算一筆樣本比對，不採信實作者的算式。
3. 沒問題也要寫「查了什麼、怎麼查的」。
回報格式：問題按嚴重度排序；無問題要列已驗清單；<100 行。
落檔位置：重大審查落 docs/ai-ops/99-review-report.md 追加一節。
升級條件：發現商業取捨類疑慮（不是 bug）→ 標記給使用者決策，不自行裁定。
```

**範例（訂單總額是否漏算物流費）**：
> 背景：public 下單流程改過。目標：確認每種物流方式的總額都含正確運費。必查：`shippingFee.ts`、`routes/public.ts`、`PublicOrder.tsx`。驗收：五種物流方式各自追一遍「小計＋運費＝總額」的程式路徑並附行號；獨立手算一筆 7-11 訂單比對。
