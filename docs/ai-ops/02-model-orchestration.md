# 02 模型調度守則

> 讀者：主對話模型（任何等級）。目的：token 花在決策，不花在搬運。

## 1. 指揮官不下場

主對話**只做**：決策、取捨、整合結論、批准升級或換路、把最終規則落到入口檔。

以下任務一律派 subagent 或分批工具，主對話不親自做：

- 大量讀取 repo（>5 檔或 >500 行）
- 掃描重複規則、檢查多檔一致性
- 搜尋 API 可用性、查網頁
- 批次改 Markdown、批次重構
- 比對 Sheet 欄位
- 驗證文件是否落地（fresh-context 審查）
- 跑測試與整理錯誤輸出

例外：主對話可以直接讀「即將要親自改的那一個檔」和「單一入口檔」。

## 2. 派工三件套

每次派工必含：**目標與動機、驗收條件、回報格式**。缺一件就是不合格派工（見 00 檔第 9 條）。模板見 `04-delegation-templates.md`。

範例：

```text
任務目標：
為了讓未來模型不再猜測成本欄位，掃描 repo 內成本相關檔案（shippingFee、pricing、fee 等），
整理現有成本欄位、公式概念與資料流。

驗收條件：
1. 每個結論附檔案路徑:行號。
2. 每個欄位標註用途。
3. 不確定的公式不得猜測，標「未確認」。
4. 輸出可直接寫入 docs/ai-ops 的 Markdown。

回報格式：
只回結論，不回長推理。每點含：結論、來源路徑:行號、風險、建議落檔位置。總長 <150 行。
```

## 3. 顯式指定 model 與 effort

**每個 session 先查當下環境實際可用什麼，不可照抄本檔。**

- 2026-07-07 本環境實查（**環境事實會過期：距此日期超過 7 天，或發現 agent 清單對不上，必須重查 session 內 system 提示，不可沿用本條**）：Agent 工具的 `model` 參數接受 `sonnet` / `opus` / `haiku` / `fable`；agent 類型有 `Explore`（唯讀搜索）、`general-purpose`（全工具）、`Plan`、`claude` 等（完整清單以 session 內 system 提示為準）。effort 參數在本環境由 agent 定義檔控制，主對話無法逐次指定（未確認未來版本；確認方法：看 Agent 工具 schema 是否有 effort 欄位）。
- 查不到可用模型時的暫定策略：低風險批次（掃檔、格式化、批次改 MD）用最便宜可用模型；高風險決策（成本公式、訂單總額）用最強可用模型；失敗後帶失敗軌跡升級。

## 4. 回報合約

subagent 回報**只能包含**：結論、來源路徑:行號、改了哪些檔案、測試結果、風險、下一步。

- 長產物必須落檔，回報只傳路徑，不把整篇塞回主對話。
- 回報超過 ~150 行 = 違約，主對話應要求重新濃縮或直接只取結論段。

## 5. 升降級路徑（硬規則）

1. 小模型（haiku 級）錯一次 → 直接升級。
2. 中階模型（sonnet 級）同一子任務連錯兩次 → 帶**完整失敗軌跡**（輸入、輸出、錯在哪）升級。
3. 高階模型解出模式後 → 把解法寫成 checklist/範例，降回便宜模型批次套用。
4. **同一件事最多重試兩輪。** 兩輪後必須換路：縮小範圍、增加驗證、改用人工確認、或明說做不到。禁止「換個措辭再試一次」式的第三次重試。

## 6. 驗證不自驗

- 文件 → read-back（重新打開檔案讀新內容）。
- 程式碼 → 跑測試或實跑；跑不了要記錄原因與替代驗證。
- 高風險判斷 → 派 fresh-context agent 第二意見，或多方案評審選優。
- **涉及成本、金流、訂單總額、物流費、客資：不允許只靠修改者自稱完成。** 最低標準：另一個 fresh-context agent 用一筆樣本資料獨立算一次比對，或使用者人工抽驗。

## 7. harness 極限（誠實條款）

拆解、驗證、多樣本評審可以補**執行**品質。但以下不能靠弱模型補：

- 模糊題（需求本身沒定義清楚）
- 產品品味與客人體驗判斷
- 商業取捨（例如犧牲毛利換衝量）

遇到這些 → 升級模型、找外部第二意見、用多方案評審、或明說做不到並停下來問使用者。
